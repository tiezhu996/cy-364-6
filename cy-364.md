## 项目主要需求

**项目概述**：连锁门店库存调配系统 —— 面向连锁零售企业，提供多门店库存统一管理、智能调拨和出入库追踪，解决门店间库存不均与缺货问题。

**项目类型**：全栈Web应用

**功能需求**：
1. **多门店SKU统一管理**：总部维护商品SKU主数据（名称、规格、条码、分类），各门店关联本地库存，支持批量导入和修改。
2. **库存实时同步与预警**：各门店库存变动实时同步，设置安全库存阈值，低于阈值时自动触发补货预警通知店长。
3. **调拨申请与审批**：门店间发起库存调拨申请，填写调拨数量和原因，目标门店确认收货后完成调拨，全程留痕可追溯。
4. **出入库记录与盘点**：记录每次入库（采购/调拨）和出库（销售/损耗）明细，支持周期性库存盘点，自动计算盘盈盘亏。
5. **滞销品分析与补货建议**：按销量排名分析各门店滞销商品，生成智能补货建议报表，辅助采购决策。

**技术要求**：
- 英文短名（用于 Docker Compose）：`ldstoreinventory`
- 前端端口：`28504`，后端端口：`29504`
- 前端框架：Vue 3 + TypeScript
- UI 库：Element Plus
- 构建工具：Vite
- 后端框架：NestJS + TypeScript
- 数据库：PostgreSQL
- 认证方式：JWT
- 其他依赖：Prisma、class-validator

## 代码存放规则

这是前后端分离项目。

前端代码放在 `frontend/`：
页面，组件，路由,状态管理,API 请求,类型定义,样式和资源等放在对应的子文件夹

后端代码放在 `backend/`：
数据库脚本放在 `database/`。

## 部署要求

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

需要提供以下文件：
- 根目录 `docker-compose.yml`：编排前端、后端、数据库及所有依赖服务
- 根目录 `.env`：包含 `COMPOSE_PROJECT_NAME=ldstoreinventory`，确保 Compose 项目名不依赖目录名（中文目录名会被规范化为空导致 `project name must not be empty` 错误）
- `frontend/Dockerfile`：前端项目构建与运行镜像（多阶段构建，先构建再用 Nginx 托管）
- `backend/Dockerfile`：后端项目构建与运行镜像
- `frontend/nginx.conf`：Nginx 配置，处理前端路由和 API 反向代理
- `.env.example`：所有环境变量的示例文件，附注释说明；同时包含 `COMPOSE_PROJECT_NAME` 默认值

docker-compose.yml 要求：
- **不要写 `version:` 字段**（Compose V2 已废弃，会产生 `the attribute version is obsolete` 警告）
- 顶层声明 `name: ldstoreinventory` 作为兜底，避免目录名为中文时项目名为空
- 所有服务使用 `container_name` 命名容器，便于管理；容器名建议带 `${COMPOSE_PROJECT_NAME:-ldstoreinventory}` 前缀
- 数据库数据使用命名卷持久化（`volumes:` 顶层声明，不要使用绑定挂载到中文路径）
- 前端通过 Nginx 反向代理后端 API（前端代码里调用 `/api`，由 Nginx 转发到 `backend:29504`，不要硬编码 `localhost`）
- 服务间通过 Docker 内部网络通信，服务名即主机名
- 数据库服务配置 `healthcheck`，后端服务通过 `depends_on: { db: { condition: service_healthy } }` 等待数据库就绪
- 后端服务也建议配置 `healthcheck`，前端 `depends_on` 后端健康
- 暴露端口：前端 `80:80`，后端按框架默认端口，数据库使用标准端口
- 所有可变配置（端口、数据库账号密码、JWT 密钥等）通过环境变量注入，并在 `.env.example` 中给出默认值
- 支持通过 `docker compose up -d` 一条命令启动全部服务，不依赖额外的 `-p` 参数

`.env` / `.env.example` 必须包含的键：
- `COMPOSE_PROJECT_NAME`：英文短名（小写字母/数字/`-`/`_`），如 `ldstoreinventory`
- 数据库连接相关：`DB_NAME` / `DB_USER` / `DB_PASSWORD` / `DB_ROOT_PASSWORD`
- 后端：`JWT_SECRET` 或等价的密钥配置
- 端口映射：`FRONTEND_PORT`、`BACKEND_PORT`、`DB_PORT`（可选，便于本地冲突时调整）

Nginx 反向代理要求（`frontend/nginx.conf`）：
- `location /api/` 反向代理到 `http://backend:29504/`
- 配置 `try_files $uri $uri/ /index.html;` 支持前端 SPA 路由
- WebSocket 场景需配置 `Upgrade` / `Connection` 头

自检清单（生成代码后必须确认）：
1. 在 **中文目录名** 下执行 `docker compose config --quiet` 不报错
2. 启动后 `docker compose ps` 显示全部容器 healthy
3. 浏览器访问前端端口能看到页面，且 `/api/...` 请求返回正常
4. README 中的 Docker 启动命令是 `docker compose up -d`（不需要 `-p`）

## docker-compose.yml 参考骨架（务必遵循）

```yaml
name: ldstoreinventory   # 顶层 name 字段兜底，避免中文目录名导致 project name 为空

services:
  db:
    image: postgres:15
    container_name: ${COMPOSE_PROJECT_NAME:-ldstoreinventory}-db
    restart: unless-stopped
    environment:
      POSTGRES_DB: ${DB_NAME}
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - db_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER} -d ${DB_NAME}"]
      interval: 5s
      timeout: 5s
      retries: 20

  backend:
    build: ./backend
    container_name: ${COMPOSE_PROJECT_NAME:-ldstoreinventory}-backend
    restart: unless-stopped
    environment:
      DB_HOST: db
      DB_PORT: 5432
      DB_NAME: ${DB_NAME}
      DB_USER: ${DB_USER}
      DB_PASSWORD: ${DB_PASSWORD}
      JWT_SECRET: ${JWT_SECRET}
    depends_on:
      db:
        condition: service_healthy

  frontend:
    build: ./frontend
    container_name: ${COMPOSE_PROJECT_NAME:-ldstoreinventory}-frontend
    restart: unless-stopped
    ports:
      - "${FRONTEND_PORT:-80}:80"
    depends_on:
      - backend

volumes:
  db_data:
```
